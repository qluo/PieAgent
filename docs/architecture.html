Pie Agent Architecture

One agent orchestrates tools. A separate face controller thread renders the current visual state. The only shared contract is FaceState.

1. Agent Design

The agent owns the conversation. Tools do one job each. The face is not a tool call; it is a shared state update that makes the agent's current activity visible.

Agent

Owns the assistant loop and decides what happens next.

  • Waits for wake word
  • Runs speech-to-text
  • Calls LLM and tools
  • Runs text-to-speech
  • Updates FaceState

2. Face Controller Design

The face controller does not talk to the LLM, microphone, or speaker. It only watches FaceState and draws the matching picture frames.

Agent Thread

Writes simple semantic states.

face_state.set("listening") face_state.set("thinking") face_state.set("speaking") face_state.set("idle")

Shared FaceState

Thread-safe shared state protected by a lock.

state = face_state.get()

Face Controller Thread

Continuously reads FaceState and asks the renderer to draw the matching animation.

  • Loads face assets
  • Chooses animation frames
  • Draws to Pi display

3. Agentic Loop

Each stage updates the face before doing slow work, so the display shows what the agent is currently doing.

Startup Wiring

This snippet shows what happens when main.py starts. It creates one shared FaceState, starts the face controller in a background thread, then starts the main agent loop in the foreground.

face_state = FaceState() face_controller = FaceController(face_state) Thread(target=face_controller.run, daemon=True).start() agent = Agent(face_state, wake_word, stt, tts, llm, tools) agent.run()
