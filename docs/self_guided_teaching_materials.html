Self-Guided Lessons to Build AI Agent on Raspberry Pi

Build a tiny AI sidekick that wakes up, listens, thinks, talks back, searches the web, and changes faces on screen.

Big Idea

One agent controls the conversation. Small tools do special jobs like wake word detection, speech-to-text, speech, local LLM answers, and search.

Starter File

Begin with demo.py. It is safe to run and easy to change.

Final Shape

Move from the demo into the clean skeleton folders: agent/, face/, agent/tools/, and faces/.

Before You Start

You Need

  • A Raspberry Pi 5 with more than 8 GB memory.
  • Python 3 installed.
  • A mini microphone.
  • A mini speaker.
  • A mini monitor or display.
  • A terminal where you can run commands.
  • uv installed for creating a Python virtual environment. Use the official uv installation instructions.
  • Ollama installed for the local LLM lesson. Use the official Ollama download and install instructions.

Helpful Rule

Make one small change at a time. Run the program. See what changed. Then make the next change.

curl -LsSf https://astral.sh/uv/install.sh | sh
uv venv
source .venv/bin/activate
uv pip install -r requirements.txt

curl -fsSL https://ollama.com/install.sh | sh
ollama pull llama3.2:1b
python3 demo.py
Lesson 1

Demo The Fake Agent

Goal: run a complete fake agent before adding microphones, speakers, real wake word detection, web search, or a local LLM.

Concepts To Explore

  • The agent loop: idle, wake, listen, think, speak.
  • Fake tools that have the same shape as real tools.
  • Why a runnable demo helps you learn the system faster.

Files And Functions

  • demo.py
  • Agent.run()
  • FakeWakeWordTool.wait()
  • FakeSpeechToTextTool.listen_and_transcribe()
  • FakeLlmTool.answer()
  • FakeTextToSpeechTool.speak()

Steps

  1. Open a terminal in the project folder.
  2. Run python3 demo.py.
  3. Press Enter when the fake wake word tool asks you to wake the agent.
  4. Type hello, what is your name?, or tell me a joke.
  5. Find FakeLlmTool.answer() and read how it chooses an answer.
  6. Add one new if rule to make the fake agent answer your own question.
  7. Run python3 demo.py again and test your new answer.

Lesson 2

Build The Face State And Face Controller

Goal: show different picture faces from the faces/ folders when the agent changes state.

Concepts To Explore

  • Shared state: the agent writes the current face state.
  • Background thread: the face controller keeps running while the agent waits.
  • Picture frames: each state loads PNG files from its matching folder.
  • Animation frames: one state can have more than one picture.

Files And Functions

  • demo.py: working face demo.
  • face/state.py: FaceState.set() and FaceState.get().
  • face/controller.py: FaceController.run().
  • face/renderer.py: load and draw PNG face pictures.
  • faces/: simple 1080 x 1080 Mini panda bot pictures. The folder idea is inspired by the original be-more-agent faces directory.

Packages And Tools

  • Image loading option: Pillow.
  • Display option: pygame for a fullscreen Raspberry Pi display.
  • Source reference for the folder/state idea: github.com/brenpoly/be-more-agent/tree/main/faces.

Steps

  1. In demo.py, find every line that calls face_state.set(...).
  2. Run the demo and watch the face state change from idle to listening, thinking, speaking, and back to idle.
  3. Open the local faces/ folder and look at the picture folders: idle, listening, thinking, speaking, error, capturing, and warmup.
  4. Open face/renderer.py. Implement load() so it reads PNG files from each folder.
  5. Store the loaded pictures in a dictionary where the key is the state name, such as "thinking".
  6. Implement draw(state) so it draws the next picture for the current state.
  7. Open face/controller.py. Implement run() so it repeatedly reads face_state.get() and asks the renderer to draw that state.
  8. Find Thread(target=face_controller.run, daemon=True).start() in main.py.
  9. Discuss why the face controller needs a background thread: the picture face should not freeze while the agent waits for input.

Lesson 4

Implement The Main Agent Loop

Goal: move the working loop idea from demo.py into the clean skeleton file.

Concepts To Explore

  • Orchestration: the agent decides what happens next.
  • Dependency injection: tools are passed into the agent instead of created inside every method.
  • State changes: the agent updates FaceState before slow work starts.

Files And Functions

  • agent/agent.py
  • Agent.run()
  • Agent.respond(user_text)
  • main.py

Steps

  1. Open demo.py and find Agent.run().
  2. Open agent/agent.py and find the skeleton Agent.run().
  3. Copy the high-level loop idea, not every fake detail.
  4. Call self.wake_word.wait() when idle.
  5. Set the face to listening, then call self.stt.listen_and_transcribe().
  6. Set the face to thinking, then call self.respond(user_text).
  7. Set the face to speaking, then call self.tts.speak(response).
  8. Set the face back to idle.
self.face_state.set("listening")
user_text = self.stt.listen_and_transcribe()

self.face_state.set("thinking")
response = self.respond(user_text)

self.face_state.set("speaking")
self.tts.speak(response)

Checkpoint

  • Agent.run() has the full loop shape.
  • The agent calls tools instead of doing tool work directly.
  • Face states match the part of the loop currently running.

Lesson 7

LLM: Add A Local Ollama Model

Goal: use a small open model running locally through Ollama.

Concepts To Explore

  • Local LLM: the model runs on your machine instead of a cloud API.
  • Prompt: the text you send to the model.
  • Response: the text the model sends back.

Files And Functions

  • agent/tools/llm.py
  • LlmTool.answer(user_text)
  • LlmTool.answer_with_context(user_text, context)
  • Agent.respond(user_text)

Packages And Tools

  • Runtime: Ollama.
  • Small model to start: llama3.2:1b.
  • Python HTTP option: requests, or Python's built-in urllib.request.
  • Local endpoint: http://localhost:11434.

Steps

  1. Install Ollama on the Raspberry Pi or development computer.
  2. Pull the model: ollama pull llama3.2:1b.
  3. Test it in the terminal: ollama run llama3.2:1b.
  4. Open agent/tools/llm.py.
  5. Implement answer(user_text) so it sends the user text to Ollama.
  6. Return only the model's answer string.
  7. Implement answer_with_context(user_text, context) later for tool results.
class LlmTool:
    def __init__(self, model_name: str = "llama3.2:1b"):
        self.model_name = model_name
        self.base_url = "http://localhost:11434"

Checkpoint

  • Ollama runs locally.
  • The LLM tool can answer a simple question.
  • The agent uses LlmTool through Agent.respond().
Lesson 8

Tools: Add Web Search

Goal: give the agent a tool it can use when it needs current information from the web.

Concepts To Explore

  • Tool routing: the agent decides when to call search.
  • Search result shaping: return a short title and snippet, not a giant page.
  • Context: search results become extra information for the LLM.

Files And Functions

  • agent/tools/search.py
  • SearchTool.search(query)
  • agent/agent.py: Agent.respond(user_text)
  • agent/tools/llm.py: answer_with_context(...)

Packages And Tools

  • Search package: duckduckgo_search.
  • Helper class: DDGS.
  • Install command: uv pip install -r requirements.txt.

Steps

  1. Open agent/tools/search.py.
  2. Install duckduckgo-search when you are ready.
  3. Import DDGS.
  4. Use ddgs.text(...) to fetch one or two results.
  5. Return a short string containing the title and snippet.
  6. Open agent/agent.py and update respond().
  7. If the user asks to “search” or “look up” something, call SearchTool.search().
  8. Send the search context into llm.answer_with_context(...).
from duckduckgo_search import DDGS

with DDGS() as ddgs:
    results = list(ddgs.text(query, region="us-en", max_results=1))

Checkpoint

  • The search tool returns a short useful result.
  • The agent can decide when to use search.
  • The LLM can answer using search context.
Lesson 9

Put Everything Together

Goal: combine the agent loop, face controller, wake word, STT, TTS, local LLM, search, and display into one Raspberry Pi assistant.

Concepts To Explore

  • Integration: connect tools that were tested separately.
  • Debugging: find which part of the loop failed.
  • Display assets: use face folders for visible states.

Files And Functions

  • main.py: wires all parts together.
  • agent/agent.py: full agent loop.
  • agent/tools/: wake word, STT, TTS, LLM, search.
  • face/: state, controller, renderer.
  • faces/: images for each face state.

Packages And Tools

  • Face rendering option: pygame for a fullscreen Pi display.
  • Face asset option: PNG images inside faces/idle, faces/listening, faces/thinking, and faces/speaking.
  • System check tools: terminal logs and small print() messages.

Steps

  1. Run each tool by itself before running the whole agent.
  2. Open main.py and check that every tool is created there.
  3. Implement FaceController.run() so it reads face_state.get().
  4. Implement FaceRenderer.load() to load image assets from the faces/ folders.
  5. Implement FaceRenderer.draw(state) to draw the next frame for the current state.
  6. Run python3 main.py.
  7. Test the full path: wake, speak question, transcribe, think, answer, return to idle.
  8. If something breaks, test the tool from that lesson again by itself.
python3 main.py

Checkpoint

  • The face changes when the agent changes state.
  • The wake word starts the interaction.
  • The agent hears, thinks, speaks, and returns to idle.
  • The search tool works when the user asks for search.
  • You can explain what each folder does.

Debugging Tips

Read The Error

The last line often tells you the real problem. Look for file names and line numbers.

Print Small Clues

Use print() to show what state, text, audio file, model answer, or search result your code has right now.

Test One Tool

If the full agent is confusing, test only one tool at a time, then reconnect it to Agent.run().

Build small. Test often. Make the agent a little smarter each lesson.
